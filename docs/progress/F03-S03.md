## Delivered

- Added PD, MR, and CQ typed registries under the existing session-owned Context service. Context and PD enforce strict child-first `EBUSY`; invalid, wrong-type, cross-session, stale, and repeated operations remain in the reviewed `EINVAL` domains.
- Added fixed-width versioned `REG_MR` payloads carrying PD identity, GPU UUID, Client address, allocation size/offset, length, access, and opaque CUDA IPC handle bytes. Responses return MR identity plus the accepted Client snapshot and direct nonzero `lkey`/`rkey`.
- Added staged MR registration and deregistration. An MR is published atomically into the identity registry, PD ownership set, and separate local/remote key indexes only after the daemon opens the CUDA mapping; close failure preserves the complete MR for retry.
- Added `resolve_lkey` and `resolve_rkey` using the QP-side PD identity, key, Client virtual range, and access bits to return the corresponding daemon address. Keys are daemon-issued, nonzero, and not reused during the process lifetime.
- Added Client CUDA allocation export using pointer attributes, allocation-range discovery, physical GPU UUID, and `cudaIpcGetMemHandle`. v1 accepts only valid `cudaMalloc` device intervals; host, managed, malformed range, unsupported handle, and invalid access cases fail before daemon publication.
- Added a daemon per-GPU runtime manager. It discovers every visible physical GPU by UUID, initializes a stable Runtime primary context per ordinal, switches to the UUID-selected device for IPC open/close, and keeps Client and daemon virtual addresses independent.
- Implemented the public PD/MR/CQ APIs without changing `include/ugdr/api.hpp`. MR snapshots preserve the Client address and expose direct keys; CQ creation enforces positive capacity, null channel, and completion vector zero; live CQ polling remains explicitly unsupported without writing WCs.
- Added fake-backend lifecycle/key/codec tests, independent-process public PD/CQ integration, and a real cross-process CUDA IPC MR smoke. Updated CMake, module boundaries, repository skeleton, and Client-visible contracts.
