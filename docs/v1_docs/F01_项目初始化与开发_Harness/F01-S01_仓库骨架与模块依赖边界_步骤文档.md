**步骤标识：** F01-S01-仓库骨架与模块依赖边界

## 一、目标与完成条件

建立与 F02-F07 模块边界一致的最小 CMake/CUDA 仓库骨架。完成时，干净工作区可配置并完成占位 target 的最小构建，实际 target 依赖与骨架文档一致，机械检查能够拒绝构造出的非法依赖。

## 二、实现设计

### 边界约束

| 类别 | 本步骤固定 | 本步骤不固定 |
|-|-|-|
| 仓库骨架 | 顶层目录、模块职责、生产 target 名称 | 运行时接口、数据结构和类层次 |
| 构建基线 | C++20、CUDA C++、CMake、Ninja、CTest | 跨环境可移植性和 CI 环境矩阵 |
| 依赖边界 | 当前占位 target 的允许链接方向及机械检查 | 后续功能新增的模块和运行时协作关系 |
| CUDA 验证 | CUDA 占位源文件能够编译 | GPU 驱动可用、`nvidia-smi` 成功或 kernel 运行成功 |
| Harness 衔接 | 边界检查器可由 Python 3 独立执行 | 完整 lint、doctor 和 smoke 入口；由后续步骤接入 |

### 文件与模块改动

| 位置 | 交付内容 | 对应 target 或规则 |
|-|-|-|
| `CMakeLists.txt` | 声明 C++20/CUDA 工程、生产 target、测试入口和 CMake File API 所需构建信息 | 全部生产 target |
| `include/ugdr` | 公共声明占位；不暴露各模块私有实现 | 公共 include 根 |
| `src/api` | Client 可见 API 门面占位源文件 | `ugdr_api` |
| `src/control` | 控制面元数据与资源管理占位源文件 | `ugdr_control` |
| `src/worker` | 数据面 Worker 占位源文件 | `ugdr_worker` |
| `src/gpu` | 可被 CUDA 编译器接受的最小占位源文件 | `ugdr_gpu` |
| `apps/client` | 最小 Client 可执行文件入口 | `ugdr_client` |
| `apps/daemon` | Control、Worker 与 GPU 的最小组合入口 | `ugdr_daemon` |
| `tests/unit`、`tests/integration`、`tests/smoke` | 保留固定测试承载目录；S01 只加入骨架与边界相关测试 | CTest 测试 target |
| `tools/module-boundaries.json` | 记录必需路径、生产 target 和允许依赖边 | 机器可读规则源 |
| `tools/check_module_boundaries.py` | 读取规则源和 CMake File API codemodel，输出边界检查结果 | 独立 Python 3 检查入口 |
| `docs/architecture/repository-skeleton.md` | 记录目录职责、target 和允许依赖方向 | 骨架说明文档 |

### 允许依赖方向

| 调用方 | 允许依赖 | 说明 |
|-|-|-|
| `ugdr_client` | `ugdr_api` | Client 只通过公共 API target 进入工程 |
| `ugdr_worker` | `ugdr_control` | Worker 仅使用控制面占位能力 |
| `ugdr_daemon` | `ugdr_control`、`ugdr_worker`、`ugdr_gpu` | Daemon 是当前骨架的组合根 |
| `ugdr_api`、`ugdr_control`、`ugdr_gpu` | 无 | 本步骤不链接其他生产 target |
| 测试 target | 被测生产 target | 生产 target 不得反向依赖测试 target |

未在表中列出的生产 target 依赖一律禁止。后续功能新增模块或依赖时，必须同步修改规则源、CMake 和骨架文档。

### 构建规则

| 规则 | 实现要求 | 失败条件 |
|-|-|-|
| 语言标准 | C++ target 使用 C++20；GPU 占位源文件由 CUDA 编译器编译 | 配置阶段无法声明要求，或编译器降级后仍报告成功 |
| 最小构建 | 所有生产 target 都有可编译占位源文件；两个应用 target 可链接 | 任一 target 缺失或构建失败 |
| 运行环境 | S01 只验证工具链编译，不执行依赖 GPU 驱动的运行时路径 | 把驱动缺失误判为骨架构建失败，或把未执行的运行时验证报告为成功 |
| 实现边界 | 公共 target 名称固定；运行时类、函数和内部实现关系暂不固化 | 占位实现提前引入后续功能决策 |

### 边界检查器

**设计伪代码：**

```python
def check_module_boundaries(root, build_dir):
    policy = load_json(root / "tools/module-boundaries.json")
    errors = []

    errors += check_required_paths(root, policy["required_paths"])
    codemodel = read_cmake_file_api(build_dir)
    errors += check_targets(codemodel, policy["production_targets"])
    errors += check_edges(codemodel, policy["allowed_edges"])
    errors += check_skeleton_doc(
        root / "docs/architecture/repository-skeleton.md",
        policy,
    )

    if errors:
        emit_each_error(errors)
        return 1
    return 0
```

| 检测条件 | 输出要求 | 退出结果 |
|-|-|-|
| 规则文件缺失、无法解析，或 CMake codemodel 不可读 | 指出文件路径和解析失败原因 | 非零 |
| 必需目录或生产 target 缺失 | 逐项列出缺失名称 | 非零 |
| 实际 target 图出现未允许依赖 | 指出调用方、被依赖方和违反规则 | 非零 |
| 骨架文档与规则源不一致 | 指出文档路径和不一致项目 | 非零 |
| 全部检查通过 | 输出明确成功结果，不混入警告性失败 | 零 |

S04 再把该检查器接入统一 `tools/ugdr lint`；S01 不提前实现完整 lint、doctor 或 smoke 流程。

## 三、验证与验收

| 顺序 | 验证动作 | 预期结果 | 失败判定 |
|-|-|-|-|
| 1 | `cmake -E make_directory build/.cmake/api/v1/query`<br>`cmake -E touch build/.cmake/api/v1/query/codemodel-v2` | 干净构建目录中存在 codemodel v2 query | 任一命令非零，或 query 文件缺失 |
| 2 | `cmake -S . -B build -G Ninja` | 配置成功；CMake File API 生成可读取的 codemodel reply | 配置非零、工具链缺失，或 codemodel reply 缺失 |
| 3 | `cmake --build build` | 全部生产 target 和 CUDA 占位源文件编译成功 | 任一约定 target 缺失或编译失败 |
| 4 | `ctest --test-dir build --output-on-failure` | 全部测试通过；负向用例构造禁止依赖，并确认边界检查器内部返回非零 | 任一测试失败，或禁止依赖未被拒绝 |
| 5 | `python3 tools/check_module_boundaries.py --root . --build-dir build` | 当前骨架通过；实际 target、允许依赖和骨架文档与规则源一致 | 命令非零，或存在缺失路径、缺失 target、未允许依赖、文档不一致 |

验证流程的外层命令必须全部返回零；负向用例只有在被测检查器返回非零且指出违反规则时，测试本身才返回零。S01 不要求 GPU 驱动或运行时 kernel 验证。
