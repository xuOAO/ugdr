## 内容边界

- 版本文档回答“这个版本为什么做、做到什么程度、包含哪些功能”。
- 功能文档回答“某项功能负责什么、如何设计边界、如何拆分步骤、如何验收”。
- 步骤文档回答“具体怎样实现和验证”。
- 不在功能文档中提前固定文件、类、函数、线程模型、数据结构字段、构建工具参数或编码步骤，除非它们是用户确认的功能级硬约束。
- 功能设计与模块边界可以描述模块职责、输入输出、控制流、数据流、关键状态变化、失败语义和与其他功能的接口边界。
- 步骤划分只定义步骤目标、交付、依赖和验收边界，不展开具体实现。步骤表后紧跟由该表生成的可渲染 Mermaid DAG；表是唯一事实源，图中边固定为“前置步骤 -> 后继步骤”，且必须包含孤立步骤。
- DAG 必须覆盖全部 `Sxx`、与表中依赖逐项一致且无环。当前剩余图中入度为 0 的步骤可以并行启动；存在循环、未知依赖或表图不一致时，相关并行结论保持“待确认”。
- 写入飞书时，DAG 必须是 Mermaid 画板（XML 为 `<whiteboard type="mermaid">...</whiteboard>` 或 `path="@*.mmd"`）。`lang=mermaid` 代码块只会显示代码，不视为可渲染飞书 DAG；Markdown 落盘时才使用 fenced `mermaid`。普通代码块、纯文本、图片或只写 DAG 标题均不合格。

3. 将版本级内容改写为功能级语句，保留来源状态：
   - 版本文档明确写出的内容标为来源事实。

   - 版本文档没有覆盖的设计，不要补成已确认结论。
4. 若版本功能划分与用户当前说法冲突，先列出冲突并请用户确认。

## 质量口径

- 目标：一至两句话说明谁能完成什么，以及如何判断实现。
- 背景：说明为什么存在、解决版本中的什么问题、承接哪些版本目标。
- 范围：写必须提供的能力、行为或交付结果。
- 非目标：写明确不处理的能力、场景或实现问题。
- 依赖与约束：写前置功能、外部环境、硬件、工具链、兼容性、性能或协作约束；未知内容标记为“待确认”。
- 功能设计：写职责、输入输出、控制流或数据流、关键状态变化、失败语义和接口边界；标明已确认和待决策内容。
- 步骤划分：每行必须包含步骤标识、步骤名称、目标与交付、依赖、验收边界；表后必须有覆盖全部步骤且与依赖列一致的无环、可渲染 Mermaid DAG。
- 验收标准：覆盖功能目标、必要质量门禁、关键错误路径、失败传播和不得发生的行为。
- 风险与待确认：用表格记录类型、内容、影响、状态。
- 变更记录：记录目标、范围、设计或步骤划分变化，以及原因和影响范围。
