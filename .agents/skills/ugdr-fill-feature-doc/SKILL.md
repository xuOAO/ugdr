# UGDR 功能文档填充

## 准备

1. 完整读取 [references/ugdr-feature-doc.md](references/ugdr-feature-doc.md)。
2. 使用飞书前执行 `lark-cli skills read lark-drive` 和 `lark-cli skills read lark-doc`，并继续读取它们要求的操作参考。
3. 先确定本轮飞书命令前缀。若用户指定 profile 或身份，则使用用户指定值；否则使用当前 lark-cli 默认身份。确定后，本轮所有 UGDR 飞书命令都使用同一个前缀，不要在执行中自动切换 profile 或身份。
4. 执行飞书读写前先运行对应前缀的 `whoami`，确认身份可用；再读取 UGDR 项目根目录、模板目录和目标版本目录验证访问权限。若 profile 缺失、token 不可用、scope 缺失或权限不足，停止并向用户报告具体错误；不要自动回退到其他身份。
5. 先按固定中文路径读取项目根目录和模板目录，确认功能模板 token 是“UGDR 文档模板”的直接子项。模板缺失、重名不明确或目录不可写时停止，不要回退到其他位置。

## 判断任务类型

- **新建功能文档**：确认版本标识、所属版本文档链接、功能标识、功能名称和来源功能项；形成稳定草稿后，创建或复用功能目录，再复制功能模板。
- **继续填写**：确认目标文档位于固定功能目录，读取现有内容后只补充或修改用户确认的章节。
- **审查功能文档**：检查功能目标、范围、边界、步骤划分、验收标准和待确认项，不写入，除非用户同时要求修改。
- **从版本文档拆出功能**：先读取版本文档的“功能划分”“版本验收标准”“风险与待确认事项”，只把已确认的版本级事实下沉为功能文档输入。
- **更新模板**：不属于默认行为；只有用户明确要求修改模板本身时才执行。

## 收集信息

按依赖顺序推进，不要一次询问所有章节：

1. 先确认所属版本、所属版本文档链接、功能标识、功能名称、功能目标和目标使用者。
2. 再确认功能背景、承接的版本目标、功能范围和非目标。
3. 收集依赖与约束；未知内容写为“待确认”，不要推测。
4. 在目标和范围稳定后整理功能设计与模块边界。描述职责、输入输出、关键状态、失败语义和接口边界，不固定文件、类、函数或逐步实现。
5. 根据功能边界提出步骤划分，由用户确认步骤目标、交付、依赖和验收边界。
6. 最后整理功能验收标准、风险、待确认事项和变更记录。

每轮只询问会改变当前章节的关键问题。先给出候选表述供用户修正，再将内容视为已确认。区分以下信息状态：

- **已确认**：用户明确表达或确认的内容，可以写入正式正文。
- **来源事实**：用户指定的现有文档中明确记录的内容，引用时说明来源。
- **建议**：Agent 提出的候选方案，未经确认不得写成项目决策。
- **待确认**：当前无法决定但需要保留的问题。

## 创建文档

仅在版本标识、所属版本文档、功能标识、功能名称和主要内容草稿得到用户确认后创建目录和文档。

1. 将版本标识规范为小写 `v<number>`；将功能标识规范为大写 `F<number>` 或用户已确认格式。无法确定时先请用户确认。
2. 将版本目录命名为 `UGDR_<版本标识> 设计`，例如 `UGDR_v1 设计`。
3. 将功能目录命名为 `<功能标识>_<功能名称>`，例如 `F01_项目初始化`。
4. 读取项目根目录，确认版本目录是项目根目录的直接子项；不存在时按用户确认创建，存在唯一同名文件夹时复用；存在同名非文件夹对象或多个同名文件夹时停止并请用户处理。
5. 读取版本目录，定位 `UGDR_<版本标识> 版本文档` 或用户确认的版本文档；所属版本文档字段必须写入该文档的飞书 docx 链接。若版本文档缺失或重名不明确，停止并请用户确认，不要写成普通文本占位。
6. 读取版本目录，确认或创建功能目录。功能文档和该功能拆出的步骤文档必须放在这个功能目录内。
7. 按飞书 Drive 规范使用 `drive files copy` 复制固定功能模板，不要通过重新创建正文模拟复制。
8. 将功能文档命名为 `<功能标识>_<功能名称> 功能文档`，除非用户明确指定其他名称。
9. 如果同名功能文档已经存在，不要覆盖或创建重复副本；先确认是继续填写现有文档还是使用新名称。
10. 创建后重新读取项目根目录、版本目录和功能目录，验证各级 `parent_token`。

禁止直接填写或覆盖模板。禁止把功能文档创建到模板目录、项目根目录、版本目录根部、Drive 根目录、“我的文档库”或其他文件夹。

## 填写文档

1. 读取目标文档当前 revision 和目标章节 block ID。
2. 使用章节级更新替换占位内容，避免全文 `overwrite`。
3. 写操作携带基准 revision，遇到并发修改时重新读取并合并，不覆盖新内容。
4. 保留模板的开头使用说明、`已完成审阅` 待办项、所属版本文档、所属版本、功能标识、功能名称、十个章节、表格结构和标题层级。
5. 所属版本文档字段必须使用飞书文档链接，显示文本可为版本文档标题；若目标版本文档无法定位，先停止确认，不要保留 `[所属版本文档链接]` 占位。
6. 只写入已确认内容；未决内容明确写为“待确认”，不要用貌似确定的措辞填满模板。
7. 每次写入后重新读取受影响章节，检查顺序、表格和语义。

`已完成审阅` 是模板正文开头的文档级人工关卡。Agent 不得自行勾选；只有用户明确确认整篇功能文档已完成审阅后，才可按用户指令将该待办改为完成。

飞书文档是人的决策载体。Agent 可以整理和建议，但不得自行扩大功能范围、确定未确认设计、引入约束或宣布验收通过。

## 质量检查

完成草稿后逐项检查：

- 功能目标说明了使用者、能力和可观察结果。
- 所属版本文档字段是链接，目标是对应版本目录下唯一确认的版本文档。
- 背景与版本关系说明该功能承接哪个版本目标或功能项。
- 功能范围与非目标没有冲突。
- 依赖与约束均有用户输入或文档来源，不包含 Agent 猜测。
- 功能设计停留在功能级，不混入步骤级文件、类、函数和编码细节。
- 每个步骤都有标识、名称、目标与交付、依赖和验收边界。
- 功能验收标准可观察、可判断，并覆盖功能目标和关键错误路径。
- 风险与待确认事项有明确状态。
- `已完成审阅` 待办项仍保留在模板开头，且未在未经用户确认时被 Agent 勾选。
- 模板占位符已被正文或“待确认”替换。
- 模板仍位于模板目录且未被修改。
- 版本目录是项目根目录的直接子项，功能目录是版本目录的直接子项，功能文档是功能目录的直接子项。

## 交付

向用户提供：

- 生成或更新后的飞书文档标题和所在路径。
- 对应版本目录和功能目录的中文路径。
- 已确认并填写的章节。
- 仍待确认的问题。
- 本次引用的来源文档。
- 文件夹位置验证结果。

在用户明确确认整篇功能文档前，将其称为“草稿”，不要宣称功能文档已经定稿。
