# UGDR 飞书文档同步到 Markdown

## 准备

1. 完整读取 [references/conversion-policy.md](references/conversion-policy.md)。
2. 读取 `lark-doc` 和 `lark-shared` skill，以及 `lark-doc` 要求的读取参考。来源由文件夹定位时再读取 `lark-drive` skill；目标是飞书原生 `.md` 文件时再读取 `lark-markdown` skill。
3. 完整读取 [`../lark-account-registry/SKILL.md`](../lark-account-registry/SKILL.md)，从仓库根目录运行账户解析脚本，并要求账户同时覆盖 `docs` 与 `drive` 用途。用户指定逻辑账户时传 `--account-id`；解析结果为 `selection_required` 时向用户展示候选并等待选择。
4. 得到 `resolved` 后，将返回的 `argv_prefix` 原样作为本轮 `<LARK>` 前缀，先运行 `<LARK> whoami`。本轮全部飞书读取复用该前缀；权限、scope、token 或资源错误时停止，不自动切换账户或默认身份。
5. 确定同步范围、目标目录和转换模式。用户未指定时，使用仓库 `docs/design/` 和 `agent` 模式。
6. 将飞书 Docx 视为权威源，将 Markdown 视为可重建的派生文档。除非用户明确要求并审阅写回内容，不要从 Markdown 回写飞书。

## 审阅门禁

任何模式、单篇同步、批量同步、差异比较或重建都必须先通过本门禁。门禁检查是本 skill 在未审阅文档上唯一允许执行的动作。

1. 定位来源后，先用 XML 仅读取文档内的审阅待办，不先读取完整正文：

   ```bash
   <LARK> docs +fetch --doc <URL_OR_TOKEN> \
     --doc-format xml \
     --scope keyword \
     --keyword "已完成审阅" \
     --detail with-ids
   ```

2. 只接受唯一且文本精确等于 `已完成审阅` 的 `<checkbox done="true">`。以下情况均视为未通过：待办未勾选、待办缺失、存在多个同名待办、返回内容无法确认 `done` 状态或读取失败。
3. 未通过时立即停止该次调用，不读取完整正文、不比较本地内容、不转换、不创建或更新任何 Markdown。提醒用户“此文档未完成审阅，请在飞书中勾选‘已完成审阅’后再同步”，并附上文档标题或链接及实际检测状态。
4. 批量同步先检查全部来源。只有每篇文档都通过后才继续；任一文档未通过时列出全部未通过项并终止整批操作，本批次不得产生部分写入。
5. 用户口头表示“已经审阅”不能替代飞书待办状态，也不能通过参数、模式或显式覆盖绕过门禁。用户勾选后必须重新读取当前状态，不使用旧缓存。

## 选择模式

- `agent`：默认。保留设计事实、边界、接口、步骤、验收、风险和状态，删除模板说明、空占位和重复叙述，供 Agent 和实现者直接使用。
- `mirror`：用于审计或归档。尽量保持原章节、正文、表格、待办和媒体顺序，只清理飞书界面噪声和无法表达的样式。
- `summary`：仅生成导航或概览，不可替代完整设计文档。保留目标、范围、关键决策、依赖、验收和未决事项。

不要把“精简”理解为概括一切。任何会影响实现、验证或责任边界的信息都必须保留。

## 定位和读取来源

1. 用户给出文档 URL 或 token 时直接使用；给出目录或版本范围时，按 `版本 -> 功能 -> 步骤` 顺序列出目标文档并建立来源清单。
2. 来源清单建立后先执行“审阅门禁”；未全部通过时不得继续本节后续步骤。
3. 通过门禁后，根据标题、所在目录和章节判断 `doc_type`：`version`、`feature` 或 `step`。证据冲突时停止该文档转换并请用户确认，不要仅凭标题猜测。
4. 每篇文档使用同一 lark-cli profile 和身份读取完整正文：

   ```bash
   <LARK> docs +fetch --doc <URL_OR_TOKEN> --doc-format markdown --detail simple
   ```

5. 从返回信封记录 `document_id`、`revision_id`、标题和原始内容。读取失败时不要用旧缓存冒充最新内容。
6. 批量同步先完成全部来源到目标路径的映射，再转换相互引用，避免生成错误的相对链接。

## 转换

1. 先应用参考中的通用保留和删除规则，再应用对应文档类型的差异化规则。
2. 保留原有决策状态。不得把“建议”“待确认”或未勾选待办改写成已确认结论。
3. 保留代码、命令、标识符、数值阈值、错误语义和验收条件的精确含义；不要为行文流畅而改动技术契约。
4. 将同批次飞书文档链接改为仓库相对链接；无法解析的引用保留原飞书 URL。
5. 按目标路径规则生成 Markdown，并写入统一 frontmatter。正文哈希仅覆盖 frontmatter 后的规范化正文。
6. `mirror` 模式需要处理图片、附件、画板或嵌入资源时，按对应飞书 skill 下载或读取；不要伪造缺失资源。`agent` 模式只本地化理解设计所必需的资源，并标注未解析引用。

## 冲突和写入

1. 目标不存在时创建新文件。
2. 目标存在且 `source_token` 不同，停止并报告路径冲突。
3. 目标存在时先运行 `python3 <SKILL_DIR>/scripts/markdown_body_hash.py --check <目标文件>`，比较现有正文哈希与 `generated_body_sha256`：
   - 一致：可用新生成内容更新。
   - 不一致：视为存在本地人工编辑，不覆盖；展示差异并请用户决定保留、合并或重建。
4. 本地正文未被修改且 `source_revision`、`content_mode` 都与目标状态相同时，报告“已是最新”，不制造无意义改动。
5. 仅在转换和链接校验完成后写入。使用 `apply_patch` 创建或更新本地文件，避免顺手改动无关文档。
6. 首次生成或更新时，可先用占位哈希写入，再运行 `python3 <SKILL_DIR>/scripts/markdown_body_hash.py <目标文件>` 得到最终正文哈希并更新 frontmatter；最后运行 `--check` 确认一致，并检查 Markdown 结构和相对链接。`<SKILL_DIR>` 是本 `SKILL.md` 所在目录。

## 批量同步

1. 先建立完整来源清单并检查每篇文档的审阅门禁；全部通过前不得开始转换或写入。
2. 门禁通过后，先同步版本文档，再同步功能文档，最后同步步骤文档。
3. 每篇文档独立记录来源 revision 和结果；单篇失败不能被汇报为整批成功。
4. 删除本地派生文档不是普通同步行为。飞书来源消失或移动时，将本地文件报告为 `stale`，只有用户明确要求时才删除或迁移。
5. 不自动生成飞书中不存在的项目决策。可以补充纯导航索引，但必须标为本地派生内容。

## 校验和交付

逐篇检查：

- frontmatter 字段完整，source token、revision、类型和模式正确。
- 标题层级连续，表格、清单、代码块和链接可读。
- 来源中唯一的 `已完成审阅` 待办为完成状态，frontmatter 的 `review_status` 为 `reviewed`；未决事项没有被提升为结论。
- 版本文档不混入步骤实现，功能文档不重复整个版本说明，步骤文档包含执行和验证所需的精确信息。
- `generated_body_sha256` 与最终正文一致。
- 批量同步的本地链接能够解析，未解析资源已明确列出。

向用户报告目标目录、转换模式、已创建/更新/跳过/冲突的文件、对应飞书 revision，以及任何未解析资源或待确认事项。
