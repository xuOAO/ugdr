# UGDR 飞书文档转 Markdown 策略

## 1. 权威源和默认方向

- 飞书 Docx 是人的协作、决策和审阅载体，也是默认权威源。
- 仓库 Markdown 是供 Agent、实现者和代码评审使用的派生快照。
- 默认同步方向是 `飞书 -> Markdown`。Markdown 中的人工修改视为提案或冲突，不自动回写飞书。
- `agent` 是默认模式；`mirror` 用于归档；`summary` 只用于导航和概览。
- 任何模式都只接受文档内唯一的 `已完成审阅` 待办已勾选的来源。未勾选、缺失、重复或状态不明时不得读取完整正文、转换、比较或写入，只提醒用户先完成飞书审阅。

## 2. UGDR 来源结构

固定项目根目录：

- 中文路径：`我的空间 / UGDR`
- folder token：`SpkxfcOjAl5fukdR9oWcSMkhnpf`

典型层级：

```text
UGDR/
└── UGDR_vN 设计/
    ├── UGDR_vN 版本文档
    └── Fxx_功能名称/
        ├── Fxx_功能名称 功能文档
        ├── Sxx_步骤名称 步骤文档
        └── ...
```

用户提供直接 URL 或 token 时不必从固定根目录重新搜索。目录定位必须验证 parent token 和唯一性；重名不明确时停止。

## 3. 默认仓库路径

用户未指定其他规则时，把飞书层级映射为：

```text
docs/design/
└── UGDR_vN/
    ├── UGDR_vN_版本文档.md
    └── Fxx_功能名称/
        ├── Fxx_功能名称_功能文档.md
        ├── Sxx_步骤名称_步骤文档.md
        └── assets/
```

文件名规则：

- 保留版本、功能和步骤标识。
- 将空格替换为 `_`，移除 `/`、NUL 和路径穿越片段；不要翻译项目名称。
- 同一来源在后续同步中保持原目标路径。标题改名不应静默移动文件。
- 用户指定现有文档目录规范时优先遵循现有规范。

## 4. Frontmatter

每个派生 Markdown 使用以下字段：

```yaml
---
source_kind: "feishu-docx"
source_token: "<document_id or stable doc token>"
source_url: "<canonical Feishu URL if known>"
source_path: "<human-readable Feishu path if known>"
source_title: "<Feishu title>"
source_revision: 12
doc_type: "version|feature|step"
content_mode: "agent|mirror|summary"
review_status: "reviewed"
synced_at: "2026-07-18T21:00:00+08:00"
generated_by: "ugdr-sync-docs-to-md"
generated_body_sha256: "<lowercase sha256>"
---
```

规则：

- 字符串一律加引号；未知 URL 或路径写空字符串，不编造。
- `review_status` 只从文本精确等于 `已完成审阅` 的唯一飞书待办得出。只有该待办已勾选才允许同步，因此新生成或更新的 Markdown 固定写 `reviewed`；不得为未勾选、缺失、重复或状态不明的来源生成 `draft` 或 `unknown` 快照。
- 正文规范化为 LF 换行并以一个换行结尾；哈希只计算 frontmatter 后的 UTF-8 正文。
- `synced_at` 使用实际同步时间和带偏移的 ISO 8601 时间。

## 5. 所有模式的通用保留规则

必须保留：

- 文档标题和有语义的章节层级。
- 目标、范围、非目标、约束、依赖和模块责任边界。
- 已确认决策、建议、待确认事项及其原始状态。
- 功能/步骤标识、接口名、文件名、命令、代码、配置键、错误码和数值阈值。
- 输入输出、控制流、数据流、状态变化、失败语义和不得发生的行为。
- 验收标准、验证命令、测试要求、质量门禁和回滚条件。
- 风险、影响、负责人或状态，以及有决策意义的变更记录。
- 有语义的表格、清单、代码块、引用和文档链接。

不得执行：

- 把未确认内容润色成确定结论。
- 为了缩短篇幅删除例外、错误路径、边界条件或验收细节。
- 更改代码、命令、路径、标识符或数值的精确含义。
- 用“同上”“详见飞书”等不可离线理解的占位语替代关键内容。

## 6. 按模式处理

### agent

删除模板使用说明、空章节、示例占位、纯排版块和无决策意义的寒暄。合并重复叙述，但保留首次完整定义和必要的来源链接。可把审阅待办映射到 frontmatter，不必在正文重复。

### mirror

保持章节顺序、正文、表格、待办、引用和媒体出现顺序。只清理飞书界面元素、不可表达的字体样式和无语义空白。审阅待办既保留在正文，也记录到 frontmatter。不要把 mirror 当作字节级备份；飞书原生布局可能无法由 Markdown 表达。

### summary

仅保留目标、范围、关键决策、核心依赖、验收摘要、主要风险和待确认事项。开头明确标注“摘要，不替代完整设计文档”，并链接来源飞书文档或同批次完整 Markdown。

## 7. 按文档类型处理

### 版本文档：导航和边界

重点保留版本目标、背景问题、范围、非目标、版本级约束、整体架构、功能划分、版本验收、风险和待确认事项。功能划分必须保留标识、职责、边界和依赖。压缩已经在功能文档中展开的设计，不保留步骤级文件、类、函数和逐条实现说明，除非它们是明确的版本级硬约束。

### 功能文档：设计契约

重点保留功能目标、版本关系、范围、非目标、依赖约束、模块边界、输入输出、关键流程、状态变化、失败语义、步骤划分、功能验收、风险和待确认事项。版本背景只保留与本功能直接相关的部分。步骤划分保留步骤标识、目标、交付、依赖和验收边界；具体命令和逐文件实现应下沉到步骤文档。

### 步骤文档：执行规范

重点保留步骤目标、前置条件、改动范围、实现约束、精确操作、文件或模块、验证命令、预期结果、验收条件、失败处理和回写要求。将长篇版本或功能背景压缩为短上下文并链接上级文档。步骤文档中的精确命令、代码片段和路径不得做语义性改写。

## 8. 链接和资源

- 同批次来源已有本地目标时，将飞书文档链接改为相对 Markdown 链接。
- 无本地目标时保留原飞书 URL，不创建猜测路径。
- `agent` 模式只下载理解设计所必需的图、画板或附件；其余保留链接并标注未本地化。
- `mirror` 模式将可下载媒体放在同功能目录的 `assets/` 下，并使用相对链接。下载失败时保留原链接并报告。
- 嵌入 Sheet、Base 或画板包含关键设计数据时，调用对应 skill 读取；不能读取时明确标注缺失，不根据缩略图猜测内容。

## 9. 增量同步和冲突

- 每次增量同步或冲突比较都先重新检查当前飞书审阅待办，不因本地已有 Markdown 或历史同步成功而跳过门禁。
- 先比较当前正文哈希；不等于 `generated_body_sha256` 时存在本地编辑，禁止自动覆盖。向用户展示来源变化和本地差异，等待合并决策。
- 当前正文未被修改且 `source_revision`、`content_mode` 均相同：默认跳过。
- 当前正文未被修改且 `source_revision` 不同：重新转换并更新。
- source token 不同但目标路径相同：路径冲突，禁止覆盖。
- 来源删除、移动或无权访问：标记为 stale 并报告；不要自动删除本地文件。
- 转换模式变化会重写正文，必须作为显式重建处理，不能伪装成普通增量同步。

## 10. 完成判定

只有满足以下条件才能报告同步成功：

- 每篇目标文档都记录了实际读取的 revision。
- 每篇来源中唯一的 `已完成审阅` 待办已勾选；批量同步不存在未通过门禁后产生的部分写入。
- 文档类型和转换模式明确。
- frontmatter 哈希与最终正文一致。
- 审阅、建议和待确认状态没有改变。
- 同批次相对链接可解析，未解析链接和资源已列出。
- 没有覆盖本地人工编辑，也没有静默删除来源缺失的文件。
