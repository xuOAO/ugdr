## 预期模板章节

当前步骤模板已按以下精简结构创建。若真实模板结构后续改动，先更新本参考再使用 skill。

模板正文开头应保留：

- 使用说明：复制本模板后，将文档放入 `UGDR_<版本标识> 设计 / <功能标识>_<功能名称>` 目录，并将标题改为 `<步骤标识>_<步骤名称> 步骤文档`。步骤文档只记录足以指导实现和验收的必要设计，不重复功能文档中的宏观目标与边界，也不为填满模板制造内容。
- 飞书待办项：`已完成审阅`，初始未勾选，位于使用说明之后、`一、目标与完成条件` 之前。
- 文档元信息：所属版本文档、所属功能文档、所属版本、功能标识、步骤标识。其中“功能标识”写为 `<Fxx>-<功能名称>`，“步骤标识”写为 `<Fxx-Sxx>-<步骤名称>`；模板不再包含独立的“功能名称”“步骤名称”或“步骤类型”字段。

`已完成审阅` 用于人工审核通过标记。Agent 不得自行勾选；只有用户明确确认整篇步骤文档已完成审阅后，才可按用户指令将其改为完成状态。

保持以下核心章节及顺序：

1. 一、目标与完成条件
2. 二、实现设计
3. 三、验证与验收
4. 四、待确认事项（可选，没有时删除）

需要时可另增“硬前置与输入”“执行流程”“回滚方案”等章节。它们不是固定章节，不适用时不要保留。

真实飞书模板是结构的唯一来源。下面的结构化写作规则只作用于“目标与完成条件”“实现设计”“验证与验收”和可选“待确认事项”的正文，不得改变模板元信息、顶层章节、章节顺序或 `已完成审阅`。除非用户明确要求更新模板本身，不得把推荐的表格或伪代码预先写进模板。

“实现设计”需要达到可直接开始编码的具体程度，但不生成固定空小节。默认表达策略如下：

| 信息类型 | 首选表达 |
|---|---|
| 文件、目录、模块、target 改动 | 表格：位置、改动、职责或影响 |
| 接口、函数、字段、数据结构 | 表格：名称、输入、输出、约束、错误 |
| 行为、状态、错误、日志 | 表格：条件、动作、状态变化、可观察结果 |
| 顺序、分支、循环、重试、并发协作 | 标注为“设计伪代码”的伪代码块 |
| 已确认约束与关键取舍 | 一至数个短段落；能放入表格时不写成长段 |
| 实现任务 | 仅在确有拆分价值时使用简短清单 |

伪代码应至少呈现与本步骤相关的输入、主要分支、状态变化、错误路径和输出。语言无关流程默认使用 Python 3 风格；若目标语言片段能更准确表达接口或资源管理，可以改用目标语言。伪代码不得虚构未确认的接口，也不得冒充已实现代码。

使用以下转换规则约束输出：

- 同一主题有三个及以上并列项、重复字段或“条件 → 动作 → 结果”对应关系时，优先使用表格。
- 表格一行只表达一个文件、接口、字段组、状态、错误或验证动作；单元格使用短句，不把原来的长段落整体搬入表格。
- 出现分支、循环、状态变化、重试、回滚或多模块协作顺序时，优先使用设计伪代码。
- 普通段落只表达一个设计点，通常控制在一至三句话；目标、约束和取舍原因可以保留为短段落。
- 表格与伪代码不得重复同一信息：表格定义静态契约，伪代码引用这些名称说明动态流程。
- 单一简单动作不强制建表；没有动态流程时不生成伪代码。

“验证与验收”存在多个动作时，优先使用“验证动作 / 预期结果 / 失败判定”表；只有单一、直接的验证动作时才用短段落或命令清单。存在前后顺序、条件分支或重试时，用伪代码表达验证编排。证据位置仅在确有记录要求时增加。

## 内容边界

- 版本文档回答“这个版本为什么做、做到什么程度、包含哪些功能”。
- 功能文档回答“某项功能负责什么、如何设计边界、如何拆分步骤、如何验收”。
- 步骤文档回答“具体怎样实现和验证”。
- 步骤文档可以记录文件、目录、模块、类、函数、CMake target、脚本命令、配置项、退出码、日志、状态记录和 CI 命令，但这些内容必须是用户确认或来源事实。
- 不在步骤文档中改变所属功能文档确认的功能目标、范围、非目标、模块边界和验收标准。
- 若实现细节反推需要修改功能边界，先停止并回到功能文档更新；不要在步骤文档中绕过上层决策。
- 规则、状态和进度分开写：规则写入实现设计或验证与验收，未决状态写入待确认事项，实际进度写入协作跟踪文档。

3. 将功能级内容改写为步骤级语句，保留来源状态：
   - 功能文档明确写出的内容标为来源事实。

   - 功能文档没有覆盖的实现细节，不要补成已确认结论。
4. 若功能文档与用户当前说法冲突，先列出冲突并请用户确认。

## 质量口径

- 目标与完成条件：一至两句话说明本步骤完成什么，以及如何判断完成。
- 模板遵从：真实飞书模板的元信息、顶层章节、章节顺序和审阅项保持不变；结构化表达只在既有章节内部使用。
- 实现设计：内容足以让实现者开始工作，不要求临场猜测关键行为。文件改动、接口契约、行为和错误规则优先用表格，不用连续大段描述替代具体设计。
- 代码级表达：存在顺序、分支、循环、状态变化、错误路径或并发协作时优先给出伪代码；伪代码注明设计示意，不冒充已经实现的代码，也不补写未确认接口。
- 验证与验收：只要求能够判断步骤是否完成；多个验证动作优先用表格关联命令或操作、预期结果和失败判定，单一简单动作不强制建表。
- 硬前置与输入：只记录真正阻断本步骤的条件，不要求完整描述步骤依赖图。
- 执行流程：仅在验证、迁移或操作类步骤存在明确顺序时使用；普通编码步骤不单独重复实现设计。
- 回滚方案：仅在存在持久化、部署或其他难以恢复的影响时使用。
- 待确认事项：只记录未解决且影响实现或验收的问题；没有时删除本节。
