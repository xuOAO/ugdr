# UGDR 步骤文档填充

## 准备

1. 完整读取 [references/ugdr-step-doc.md](references/ugdr-step-doc.md)。
2. 使用飞书前执行 `lark-cli skills read lark-drive` 和 `lark-cli skills read lark-doc`，并继续读取它们要求的操作参考。
3. 先确定本轮飞书命令前缀。若用户指定 profile 或身份，则使用用户指定值；否则使用当前 lark-cli 默认身份。确定后，本轮所有 UGDR 飞书命令都使用同一个前缀，不要在执行中自动切换 profile 或身份。
4. 执行飞书读写前先运行对应前缀的 `whoami`，确认身份可用；再读取 UGDR 项目根目录、模板目录、目标版本目录和目标功能目录验证访问权限。若 profile 缺失、token 不可用、scope 缺失或权限不足，停止并向用户报告具体错误；不要自动回退到其他身份。
5. 先按固定中文路径读取项目根目录和模板目录，确认步骤模板 token 是“UGDR 文档模板”的直接子项。步骤模板缺失、重名不明确或目录不可写时停止，不要用功能模板、版本模板或手写正文模拟步骤模板。
6. 真实飞书步骤模板是文档结构的唯一来源。创建或填写前读取模板当前结构；本 skill 的表格和伪代码规则只控制核心章节内部的表达，不得据此改写模板元信息、顶层章节、章节顺序或审阅项。

## 判断任务类型

- **新建步骤文档**：确认所属版本文档、所属功能文档、功能标识及名称、步骤标识及名称和来源步骤项；形成稳定草稿后，在对应功能目录内复制步骤模板。
- **继续填写**：确认目标步骤文档位于固定功能目录，读取现有内容后只补充或修改用户确认的章节。
- **审查步骤文档**：检查目标与完成条件、实现设计、验证与验收和待确认项是否足以指导工作，不按固定子项检查，不写入，除非用户同时要求修改。
- **从功能文档拆出步骤**：先读取功能文档的“七、步骤划分”“八、验证与功能验收标准”“九、风险与待确认事项”，只把已确认的功能级事实下沉为步骤文档输入。
- **更新模板**：不属于默认行为；只有用户明确要求修改模板本身时才执行。

## 收集信息

按依赖顺序推进，不要一次询问所有章节：

1. 先确认所属版本、所属版本文档链接、所属功能文档链接、功能标识及名称、步骤标识及名称，以及目标与完成条件。模板把名称合并写入“功能标识”和“步骤标识”，不再单设“功能名称”“步骤名称”或“步骤类型”字段。
2. 只收集会阻断本步骤的硬前置、输入和功能边界；不要要求穷举前后步骤关系。少量关联修改可以写入实现设计，只有改变功能边界时才回到功能文档确认。
3. 以实现设计为主体，优先收集可直接落实到文件、模块、接口、数据结构、行为、流程、错误处理和验证动作的具体信息。已确认路径、符号、target、字段、状态和命令应尽量写出，不用抽象长段落代替可执行设计。
4. 按信息类型选择表达形式：文件与模块改动、接口契约、状态或错误、验证项优先用表格；控制流、状态转换、并发协作或脚本编排优先用伪代码；只有目标、约束、取舍原因和表格无法承载的语义使用短段落。形式按需选用，不为简单步骤制造空表格或无意义伪代码。
5. 整理与复杂度相称的验证与验收方式。存在多项命令或判定时，优先用“验证动作 / 预期结果 / 失败判定”表；存在顺序、分支或重试时，用伪代码补充。只要求文档能够判断本步骤是否完成。
6. 最后只保留仍会影响实现或验收的待确认事项。硬前置与输入、执行流程、回滚方案仅在确有必要时另增章节。

每轮只询问会改变当前章节的关键问题。先给出候选表述供用户修正，再将内容视为已确认。区分以下信息状态：

- **已确认**：用户明确表达或确认的内容，可以写入正式正文。
- **来源事实**：用户指定的现有文档中明确记录的内容，引用时说明来源。
- **建议**：Agent 提出的候选方案，未经确认不得写成实现决策。
- **待确认**：当前无法决定但需要保留的问题。

## 创建文档

仅在版本标识、所属版本文档、所属功能文档、功能标识及名称、步骤标识及名称和主要内容草稿得到用户确认后创建文档。

1. 将版本标识规范为小写 `v<number>`；将功能标识规范为大写 `F<number>`；将步骤标识规范为大写 `F<number>-S<number>` 或用户已确认格式。无法确定时先请用户确认。
2. 将版本目录命名为 `UGDR_<版本标识> 设计`，例如 `UGDR_v1 设计`。
3. 将功能目录命名为 `<功能标识>_<功能名称>`，例如 `F01_项目初始化`。
4. 读取项目根目录，确认版本目录是项目根目录的直接子项；不存在时停止并请用户确认是否应先创建版本目录，不要自动创建不明确版本。
5. 读取版本目录，定位 `UGDR_<版本标识> 版本文档` 或用户确认的版本文档；所属版本文档字段必须写入该文档的飞书 docx 链接。
6. 读取版本目录，定位唯一的功能目录；读取功能目录，定位 `<功能标识>_<功能名称> 功能文档` 或用户确认的功能文档。所属功能文档字段必须写入该文档的飞书 docx 链接。
7. 按飞书 Drive 规范使用 `drive files copy` 复制固定步骤模板，不要通过重新创建正文模拟复制。
8. 将步骤文档命名为 `<步骤标识>_<步骤名称> 步骤文档`，例如 `F01-S01_仓库骨架与模块依赖边界 步骤文档`，除非用户明确指定其他名称。
9. 如果同名步骤文档已经存在，不要覆盖或创建重复副本；先确认是继续填写现有文档还是使用新名称。
10. 创建后重新读取项目根目录、版本目录和功能目录，验证各级 `parent_token`。

禁止直接填写或覆盖模板。禁止把步骤文档创建到模板目录、项目根目录、版本目录根部、Drive 根目录、“我的文档库”或其他文件夹。

## 填写文档

1. 读取目标文档当前 revision 和目标章节 block ID。
2. 使用章节级更新替换占位内容，避免全文 `overwrite`。
3. 写操作携带基准 revision，遇到并发修改时重新读取并合并，不覆盖新内容。
4. 保留模板的开头使用说明、`已完成审阅` 待办项、所属版本文档、所属功能文档、所属版本、功能标识、步骤标识，以及“目标与完成条件”“实现设计”“验证与验收”三个核心章节。“功能标识”写为 `<Fxx>-<功能名称>`，“步骤标识”写为 `<Fxx-Sxx>-<步骤名称>`；不要新增已从模板移除的独立名称或步骤类型字段。
5. 所属版本文档和所属功能文档字段必须使用飞书文档链接，显示文本可为对应文档标题；若目标文档无法定位，先停止确认，不要保留链接占位。
6. 只写入已确认内容；未决内容明确写为“待确认”，不要用貌似确定的措辞填满模板。表格和伪代码只放在模板已有核心章节内部；不为采用这些形式而新增固定顶层章节，也不修改模板本身。
7. 同一主题出现三个及以上并列项、重复字段或条件与结果对应关系时，优先转为表格。表格一行表达一个文件、接口、状态、错误或验证动作，单元格保持短句；不要把长段落原样塞进单元格。
8. 关键行为出现分支、循环、状态变化、重试、回滚或多模块协作顺序时，优先提供标注为“设计伪代码”的伪代码。伪代码表达输入、主要路径、错误路径和输出，不得冒充已实现代码；语言无关流程默认使用易读的 Python 3 风格，目标语言更准确时可使用目标语言片段。
9. 普通段落原则上只表达一个设计点，并控制在一至三句话。连续解释文件改动、接口字段、行为矩阵、错误规则或验收判定时，必须重新判断能否改为表格或伪代码。
10. 实现设计和验证与验收内部不保留不适用的空小节、空表格或占位项。每次写入后重新读取受影响章节，检查模板结构、表达形式和语义。

`已完成审阅` 是模板正文开头的文档级人工关卡。Agent 不得自行勾选；只有用户明确确认整篇步骤文档已完成审阅后，才可按用户指令将该待办改为完成。

步骤文档是实现执行前的决策载体。Agent 可以整理和建议，但不得自行扩大功能范围、改变已确认边界、引入未确认工具版本或宣布验证通过。

## 质量检查

完成草稿后逐项检查：

- 目标与完成条件说明了本步骤要完成什么，以及如何判断完成。
- 所属版本文档和所属功能文档字段都是飞书文档链接，且目标位于固定目录层级。
- 文档保留真实飞书模板的元信息、顶层章节、章节顺序和审阅项；表格、伪代码与必要的小节只出现在模板核心章节内部。
- 实现设计提供了开始工作所需的关键决策，未要求实现者临场猜测重要行为；可结构化的信息已优先使用表格，存在顺序、分支、状态变化或错误路径的关键流程已优先使用伪代码。
- 正文没有用连续大段描述代替文件改动、接口契约、行为矩阵、错误处理或验证判定；普通段落只表达一个设计点且通常不超过三句话。
- 实现设计没有超出所属功能文档确认的边界；少量关联修改已说明，改变功能边界的内容已回到功能文档确认。
- 文件、命令、target、退出码、日志和状态规则等具体内容均有用户确认或来源事实。
- 验证与验收能够判断步骤是否完成；存在多个验证动作时已使用表格关联动作、预期结果和失败判定，不把它们拆散在长段落中。
- 若保留待确认事项，该节只包含尚未解决且影响实现或验收的问题，规则、状态和实际进度没有混写。
- `已完成审阅` 待办项仍保留在模板开头，且未在未经用户确认时被 Agent 勾选。
- 核心章节的模板占位符已被正文或“待确认”替换；不适用的可选内容已删除。
- 模板仍位于模板目录且未被修改。
- 版本目录是项目根目录的直接子项，功能目录是版本目录的直接子项，步骤文档是功能目录的直接子项。

## 交付

向用户提供：

- 生成或更新后的飞书文档标题和所在路径。
- 对应版本目录、功能目录和步骤文档的中文路径。
- 已确认并填写的章节。
- 仍待确认的问题。
- 本次引用的来源文档。
- 文件夹位置验证结果。

在用户明确确认整篇步骤文档前，将其称为“草稿”，不要宣称步骤文档已经定稿。
