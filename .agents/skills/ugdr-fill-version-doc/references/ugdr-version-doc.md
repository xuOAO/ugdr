## 内容边界

- 版本文档回答“这个版本为什么做、做到什么程度、包含哪些功能”。
- 功能文档回答“某项功能负责什么、如何验收”。
- 步骤文档回答“具体怎样实现和验证”。
- 不在版本文档中提前确定接口、文件、类、函数或编码步骤，除非它们是已确认的版本级硬约束。
- “七、功能划分”先用功能表记录 `Fxx`、职责、边界和依赖，再紧跟由该表生成的可渲染 Mermaid DAG。表是唯一事实源，图中边固定为“前置功能 -> 后继功能”，且必须包含孤立功能。
- DAG 必须覆盖全部功能、与表中依赖逐项一致且无环。当前剩余图中入度为 0 的功能可以并行启动；存在循环、未知依赖或表图不一致时，相关并行结论保持“待确认”。
- 写入飞书时，DAG 必须是 Mermaid 画板（XML 为 `<whiteboard type="mermaid">...</whiteboard>` 或 `path="@*.mmd"`）。`lang=mermaid` 代码块只会显示代码，不视为可渲染飞书 DAG；Markdown 落盘时才使用 fenced `mermaid`。普通代码块、纯文本、图片或只写 DAG 标题均不合格。

所有 UGDR 飞书操作都先确定本轮命令前缀：

```bash

```

继续前必须确认 `whoami` 返回身份可用，并用同一个 `<LARK>` 前缀读取 UGDR 项目根目录。后续命令必须复用同一个 `<LARK>` 前缀和同一个身份策略。若缺少 scope、文件夹权限或文档编辑权限，停止并报告错误；不要自动改用其他 profile 或身份。
