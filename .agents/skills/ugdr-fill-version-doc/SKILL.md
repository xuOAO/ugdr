# UGDR 版本文档填充

## 准备

1. 完整读取 [references/ugdr-version-doc.md](references/ugdr-version-doc.md)。
2. 使用飞书前执行 `lark-cli skills read lark-drive` 和 `lark-cli skills read lark-doc`，并继续读取它们要求的操作参考。
3. 完整读取 [`../lark-account-registry/SKILL.md`](../lark-account-registry/SKILL.md)，从仓库根目录运行账户解析脚本，并要求账户同时覆盖 `docs` 与 `drive` 用途。用户指定逻辑账户时传 `--account-id`；用户只给出未登记的 raw profile 时先停止并按账户 Skill 登记，不直接绕过注册表。
4. 若解析结果为 `selection_required`，向用户展示候选并等待选择；得到 `resolved` 后，将返回的 `argv_prefix` 原样作为本轮 `<LARK>` 前缀。本轮所有 UGDR 飞书命令都使用同一个前缀，不持久化选择，也不自动切换账户。
5. 执行飞书读写前先运行 `<LARK> whoami`，确认身份可用；再读取 UGDR 项目根目录验证访问权限。若 profile 缺失、token 不可用、scope 缺失或权限不足，停止并向用户报告具体错误；不要自动回退到其他账户或默认身份。
6. 先按固定中文路径读取项目根目录和模板目录，确认模板 token 是“UGDR 文档模板”的直接子项。模板缺失、重名不明确或目录不可写时停止，不要回退到其他位置。

## 判断任务类型

- **新建版本文档**：访谈并形成稳定草稿，确认版本标识和文档名后，创建或复用版本设计文件夹，再复制模板。
- **继续填写**：确认目标文档位于固定文件夹，读取现有内容后只补充或修改用户确认的章节。
- **审查版本文档**：检查内容质量和待确认项，不写入，除非用户同时要求修改。
- **更新模板**：不属于默认行为；只有用户明确要求修改模板本身时才执行。

## 收集信息

按依赖顺序推进，不要一次询问所有章节：

1. 先确认版本标识、版本目标和目标使用者。
2. 再确认背景、核心问题、版本范围和非目标。
3. 收集已确定的约束；未知内容写为“待确认”，不要推测。
4. 在目标和范围稳定后整理宏观架构。只描述模块及关系，不提前设计接口或步骤实现。
5. 根据范围和架构提出功能划分，由用户确认功能职责、边界和依赖；再从已确认依赖生成覆盖全部功能的 DAG。
6. 最后整理版本验收标准、风险、待确认事项和变更记录。

每轮只询问会改变当前章节的关键问题。先给出候选表述供用户修正，再将内容视为已确认。区分以下信息状态：

- **已确认**：用户明确表达或确认的内容，可以写入正式正文。
- **来源事实**：用户指定的现有文档中明确记录的内容，引用时说明来源。
- **建议**：Agent 提出的候选方案，未经确认不得写成项目决策。
- **待确认**：当前无法决定但需要保留的问题。

### 功能依赖 DAG

- “七、功能划分”中的功能表是依赖关系的唯一事实源，DAG 只由该表生成，不单独维护第二套依赖。
- DAG 紧跟功能表并留在原章节内。节点使用稳定的 `Fxx` 标识并附简短名称，孤立节点也必须出现。
- 边方向固定为 `前置功能 -> 后继功能`。当前剩余图中入度为 0 的功能属于同一可并行前沿；功能完成后重新计算前沿。
- 图必须覆盖功能表中的全部节点，且不得包含表外节点、遗漏依赖或形成环。发现循环、未知依赖或表图不一致时，标记“待确认”，不得宣称相关功能可并行。
- 使用可渲染 Mermaid `flowchart LR` 表达。写入飞书时必须创建 Mermaid 画板（XML 为 `<whiteboard type="mermaid">...</whiteboard>` 或 `path="@*.mmd"`），不得用 `lang=mermaid` 代码块、普通代码块、纯文本、图片或只写“DAG”标题替代；Markdown 落盘时才使用 fenced `mermaid` 代码块。

## 创建文档

仅在版本标识、文档名称和主要内容草稿得到用户确认后创建目录和文档。

1. 将版本标识规范为小写 `v<number>`；无法按此格式确定时先请用户确认。
2. 将版本目录命名为 `UGDR_<版本标识> 设计`，例如 `UGDR_v1 设计`。
3. 读取项目根目录：存在唯一同名文件夹时复用；不存在时使用 `drive +create-folder` 在项目根目录下创建；存在同名非文件夹对象或多个同名文件夹时停止并请用户处理。
4. 按飞书 Drive 规范使用 `drive files copy` 复制固定模板，不要通过重新创建正文模拟复制。
5. 将版本文档命名为 `UGDR_<版本标识> 版本文档`，除非用户明确指定其他名称。
6. 将版本文档直接创建在对应版本目录中。如果同名文档已经存在，不要覆盖或创建重复副本；先确认是继续填写现有文档还是使用新名称。
7. 创建后重新读取项目根目录和版本目录，验证版本目录及文档的 `parent_token`。

禁止直接填写或覆盖模板。禁止把版本文档创建到模板目录、项目根目录、Drive 根目录、“我的文档库”或其他文件夹。

版本目录是该版本设计文档的统一归档位置。版本文档直接放在版本目录下。归属于该版本的功能文档和步骤文档不得散落在项目根目录或版本目录根部；每个功能必须先在版本目录下创建一个功能目录，功能文档和该功能拆出的步骤文档都放在对应功能目录内。

功能目录默认命名为 `<功能标识>_<功能名称>`，例如 `F01_项目初始化`。功能文档默认命名为 `<功能标识>_<功能名称> 功能文档`；步骤文档默认命名为 `<步骤标识>_<步骤名称> 步骤文档`。功能标识、功能名称或步骤标识不明确时先请用户确认，不要自行编造。

## 填写文档

1. 读取目标文档当前 revision、目标章节 block ID，以及文本精确等于 `已完成审阅` 和 `已实现` 的待办 block ID 与 `done` 状态。`已完成审阅` 缺失、重名或状态不可判定时，在任何正文写入前停止；`已实现` 缺失时先在 `已完成审阅` 后补一个未勾选待办并重新读取，重名或状态不可判定时停止。
2. 使用章节级更新替换占位内容，避免全文 `overwrite`。
3. 写操作携带基准 revision，遇到并发修改时重新读取并合并，不覆盖新内容。
4. 保留模板的开头使用说明、`已完成审阅` 与 `已实现` 待办项、十个章节、表格结构和标题层级。
5. 只写入已确认内容；未决内容明确写为“待确认”，不要用貌似确定的措辞填满模板。
6. 每次写入后重新读取受影响章节，检查顺序、表格和语义。

### 人工状态失效处理

1. 对既有目标文档首次成功执行会改变决策内容的写入后，如果写入前 `已完成审阅` 为 `done=true`，立即重新读取最新 revision 和该待办 block，并用 block 级更新将它改为 `done=false`；保留待办文本、富文本样式和所在位置。重置旧审阅结论不需要额外征求用户确认。
2. 如果写入影响版本目标、范围、功能划分、验收依据、依赖 DAG 或其他实现依据，且写入前 `已实现` 为 `done=true`，同样立即重置为 `done=false`；保留待办文本、富文本样式和所在位置。只修正错别字、格式或不影响实现依据的链接展示时可不重置 `已实现`。
3. 会触发状态重置的写入包括正文、元信息、章节、表格、链接、Mermaid DAG 或其他影响文档语义与验收依据的变化。只读审查、无实际变化的操作，以及用户明确要求把对应待办改为 `done=true` 的动作不触发反向重置。
4. 同一轮有多次写入时，在首次内容写入成功后立刻完成必要重置，不等到全部写入结束。重置失败时停止后续写入，报告“正文已修改但人工状态重置失败”，且不得把该文档同步为已审阅 Markdown。
5. 交付前再次按关键词读取两个待办。新建或本轮修改过的文档必须存在唯一 `已完成审阅` 和唯一 `已实现`；除用户本轮明确完成整篇审阅或确认版本已实现外，对应待办最终状态必须为 `done=false`。

`已完成审阅` 和 `已实现` 是模板正文开头的文档级人工关卡，位于使用说明之后、`一、版本目标` 之前。Agent 不得自行勾选；只有用户明确确认整篇版本文档已完成审阅或该版本已实现后，才可按用户指令将对应待办改为完成。任何后续内容修改都会使相关人工结论失效，并按上述规则自动重置。

飞书文档是人的决策载体。Agent 可以整理和建议，但不得自行扩大版本范围、确定架构、引入约束或宣布验收通过。

## 质量检查

完成草稿后逐项检查：

- 版本目标说明了使用者、能力和可观察结果。
- 版本范围与非目标没有冲突。
- 约束均有用户输入或文档来源，不包含 Agent 猜测。
- 整体架构停留在版本级，不混入步骤级实现。
- 每个功能都有标识、名称、职责、边界和依赖。
- 功能依赖 DAG 与功能表逐项一致、无环，并能识别当前可并行前沿；飞书 DAG 必须是 Mermaid 画板，Markdown DAG 必须是 fenced `mermaid` 代码块。
- 验收标准可观察、可判断，并覆盖版本目标。
- 风险与待确认事项有明确状态。
- `已完成审阅` 和 `已实现` 待办项仍唯一保留在模板开头；本轮修改过的既有文档已按“人工状态失效处理”重置并复查，且未在未经用户确认时被 Agent 勾选。
- 模板占位符已被正文或“待确认”替换。
- 模板仍位于模板目录且未被修改。
- 版本目录是项目根目录的直接子项，名称符合 `UGDR_<版本标识> 设计`。
- 版本文档是对应版本目录的直接子项。
- 每个功能目录是对应版本目录的直接子项。
- 功能文档和该功能的步骤文档是对应功能目录的直接子项。

## 交付

向用户提供：

- 生成或更新后的飞书文档标题和所在路径。
- 对应版本目录的中文路径。
- 已确认并填写的章节。
- 仍待确认的问题。
- 当前可并行启动的功能；若 DAG 尚未通过一致性检查，明确说明原因。
- 本次引用的来源文档。
- 文件夹位置验证结果。
- `已完成审阅` 和 `已实现` 的最终状态；若本轮修改使其重置，明确提醒用户需要重新审阅或重新确认实现状态，重新审阅前不得同步为已审阅 Markdown。

在用户明确确认整篇版本文档前，将其称为“草稿”，不要宣称版本文档已经定稿。
